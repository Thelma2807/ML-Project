# Columns common to IMUs (some files do not have everything)
COMMON = ['time','accX','accY','accZ','gyroX','gyroY','gyroZ','magX','magY','magZ','name']

def read_common(path, nrows=None):
    """Loads our CSV and keeps only known IMU columns + source."""
    df = pd.read_csv(path, nrows=nrows)
    keep = [c for c in COMMON if c in df.columns]
    df = df[keep].copy()
    for c in df.columns:
        if c not in ('time','name'):
            df[c] = pd.to_numeric(df[c], errors='coerce')
    df['source'] = Path(path).name
    return df

def add_norms(df):
    """Add acc/gyro/mag standards if possible."""
    for name, cols in {
        'acc_norm':  ['accX','accY','accZ'],
        'gyro_norm': ['gyroX','gyroY','gyroZ'],
        'mag_norm':  ['magX','magY','magZ'],
    }.items():
        have = [c for c in cols if c in df.columns]
        if len(have)==3:
            df[name] = np.sqrt((df[have]2).sum(axis=1))
    return df

# Robust estimation of sample rate per file
def estimate_hz(df):
    if 'time' not in df:
        return np.nan
    #Float test (timestamps in seconds)
    ts = pd.to_numeric(df['time'], errors='coerce')
    if ts.notna().sum() >= 3:
        d = np.diff(ts.dropna().values)
        d = d[d>0]
        if len(d):
            med = np.median(d)
            return round(1.0/med, 3) if med>0 else np.nan
    #Datetime Test
    t = pd.to_datetime(df['time'], errors='coerce', utc=True, infer_datetime_format=True)
    if t.notna().sum() >= 3:
        d = np.diff(t.dropna().astype('int64').values/1e9)
        d = d[d>0]
        if len(d):
            med = np.median(d)
            return round(1.0/med, 3) if med>0 else np.nan
    return np.nan

# Statistical windowing (features on sliding windows)
WIN = 50          #50 samples
OVERLAP = 0.5     # 50% recovery

def window_features(df, cols, win=WIN, overlap=OVERLAP):
    step = max(1, int(win*(1-overlap)))
    N = len(df)
    rows = []

        end = start + win
        sl = df.iloc[start:end]

3 - Formalisation of the problem

1) Context & Objective

IMU measurements are provided for the UR3e robotic arm, recorded during normal operations such as screw driving, painting, and pick-and-place, as well as during simulated anomalies including earthquake, hitting the arm or platform, added weight, and magnetic interference. The key objective is to develop a TinyML-compatible approach that will result in the detection of these anomalies in real time and the generation of reliable alerts with low computational overhead and minimum latency.

Machine learning formulation binary classification per window

  • Input: Statistical features extracted from sliding windows over IMU time series.
  • Output: y {0,1}, where 0=normal, 1=anomaly.

2) Observation Unit (Time Windowing)

The input data files are sampled at various frequencies, that is, 2/10/20 Hz. For consistency in our analysis, we work with fixed-size windows of W samples (default: W = 50). This gives approximate time durations as follows:

  • 20 Hz 50 points 2.5 seconds
  • 10 Hz 50 points 5 seconds
  • 2 Hz 50 points 25 seconds (longer window that could be reduced if necessary)

Note: Overlapping windows (e.g., 50%) can increase granularity without increasing labeling costs.

Non-overlapping window segmentation (for the baseline): If a file contains N rows, then the number of windows is N / W.

3) Input Variables (Features)

From the (X, Y, Z) axes, we compute the vector norms:

  • acc_norm = (accX + accY + accZ)
  • gyro_norm = (gyroX + gyroY + gyroZ)
  • mag_norm = (magX + magY + magZ)

For each window, we extract robust statistical features from each selected signal: axes and/or norms:

  • Mean, standard deviation, min, max, RMS ((1/W * s))

These features are computationally efficient, sensor-agnostic, and well-suited for TinyML.

4) Target Variable (Labels)

There is no explicit label column in the files, so we rely on file naming conventions:

  • label = 1 if filename contains: earthquake, hitting_arm, hitting_platform, extra_weight, magnet
  • label = 0 otherwise, i.e., for normal IMU files like IMU_2Hz, IMU_10Hz, IMU_Acc_Gyro_20Hz, etc.

Window label (simple majority rule): If 50% of the points in a window are labeled 1, then the window is labeled 1; otherwise, it's 0. This helps in smoothing out transient noise.

5) Selected Preprocessing Steps

  • Robust numerical casting (errors='coerce'), and removal of rows which are completely empty across the IMU columns.
  • Per-source standardization (by file) to handle sensor offsets: z = (x - _source) / _source This normalization is computed separately for each file to prevent data leakage.
  • Window aggregation statistical feature extraction, as explained above.

6) Evaluation Strategy & Data Leakage Prevention

Samples from the same file are highly correlated. So evaluation needs to be done by file, not by random splitting of the rows.

  • Split strategy: GroupShuffleSplit with group = source (i.e., filename). Test 30% of the files, Train 70%

  • More robust variant: Leave-One-Group-Out (LOGO), where each file is tested in turn (more expensive but informative).

7) Evaluation Metrics

  • ROC-AUC (global): to measure overall ranking ability
  • PR-AUC/F1: to account for class imbalance (anomalies are rare)
  • Confusion matrix (TP/FP/FN/TN): to set realistic operational thresholds, e.g., acceptable FP rate vs. target recall

Example industrial targets:

  • Recall (anomalies) 95%
  • False positive rate 2% (adjustable depending on the context)

8) Class Imbalance Management

  • Windowing and majority-rule already reduce the extreme point-wise rarity.
  • During training: class_weight='balanced' or oversampling/undersampling of windows.
  • For evaluation: always stratify by file and report PR-AUC.

9) Baseline Model (Step 4 Preview)

  • Random Forest (200 trees) on statistical features: robust, low tuning effort, interpretable via feature importance.

TinyML-Friendly Alternatives:

  • Logistic Regression (low memory footprint)
  • Small trees / slant (oblique) trees (14 nodes)
  • Isolation Forest: unsupervised, handy when labels are unreliable

10) Real-Time & Embedded Constraints

  • Latency window size / frequency (e.g., 50 points @ 20 Hz 2.5 seconds)

  • If latency must be < 1s, two options:

    1. Reduce W (e.g., 2030) for 10/20 Hz files
    2. Use overlapping windows (smaller stride) to increase decision density without perceived latency increase

  • Computation cost: statistical features are O(W), lightweight models (LR/mini-tree) can run on MCUs

11) Success Criteria (Baseline)

  • Fully reproducible pipeline (same splits, same per-source normalization)
  • ROC-AUC 0.90 and reasonable PR-AUC on per-file test set
  • ROC/PR curves + confusion matrix + top feature importances
  • Save model and normalization parameters (per source if needed)

12) Limitations & Risks

  • File-based labeling is approximate; mixed segments may still exist. Majority voting helps, but doesn't eliminate our problem
  • Heterogeneous frequencies mean that fixed-size windows span different durations. Eventually use fixed durations (in seconds) instead of fixed sample counts
  • Covariate shift between tasks: for instance, screwdriving versus painting. Validate using LOGO and consider task-specific features or domain labels.
